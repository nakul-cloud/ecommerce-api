# `app/schemas/` — Data Validation Layer (Pydantic Models)

This folder contains schema definitions that validate request inputs and format response outputs.

---

## 1. Purpose

> Why do we have a schemas folder?

Pydantic schemas enforce type-safety and structural rules at the API boundary:
- **Automatic request validation**: Ensure the client provides valid data structures (e.g. valid string lengths, non-negative numbers) before execution.
- **Data sanitization**: Filter out private database fields (like `cost_price`) from API responses automatically.
- **Autogenerated documentation**: Populate request/response structures in Swagger UI automatically.

Without schemas, you would have to write manual `if/else` checks for every field of every incoming request.

---

## 2. Responsibilities

### What belongs inside `schemas/`

- Pydantic model definitions (`BaseModel`).
- Validation rules (`Field(gt=0, min_length=3)`).
- Input-specific structures (`ProductCreate`).
- Output-specific structures (`ProductResponse`).

### What does NOT belong inside `schemas/`

- Database queries or configurations.
- Business calculations or logic.

---

## 3. Files

### `product_schema.py`

- **Purpose**: Validates product inputs and formats responses.
- **Classes**:
  - `ProductCreate`: Validates attributes needed for creation (`name`, `price`, `stock_quantity`, `cost_price`).
  - `ProductUpdate`: Allows patch updates with optional fields.
  - `ProductResponse`: Fields to return (`id`, `name`, `description`, `category`, `price`, `stock_quantity`). Excludes `cost_price`.

### `order_schema.py`

- **Purpose**: Defines order request and response payloads.
- **Classes**:
  - `OrderItem`: Represens `product_id` and `quantity`.
  - `OrderCreate`: Contains a list of `OrderItem` objects.
  - `OrderResponse`: Returns `id`, `total_amount`, and items.

---

## 4. Request Flow

Schemas run at the very entry and exit points of the router handler execution:

```
Client ──► [Schema (Validate Input)] ──► Route ──► Controller ──► DB
                                                                   │
                                                                   ▼
Client ◄── [Schema (Format Output)] ◄─── Route ◄─── Controller ◄───┘
```

Schemas are like application forms. They specify which fields are required, their data types, and constraint rules (e.g., "Age must be positive"). If a form is filled out incorrectly, the receptionist rejects it immediately. If it's valid, it gets passed to the manager.

## 7. Best Practices

### Do

- Keep input schemas (`Create`/`Update`) separate from response schemas.
- Use strict fields validators (`gt=0`, `min_length`).
- Document fields using `description` inside `Field`.

### Don't

- Never include sensitive fields (like API keys or cost prices) in Response schemas.
- Avoid writing business rules directly inside schema validators.

---

## 8. Interview Questions

1. **Difference between `ProductCreate` and `ProductResponse`?**
   `ProductCreate` includes validation for inputs, including private variables like `cost_price`. `ProductResponse` defines what is returned, hiding `cost_price` and adding database-generated fields like `id`.
2. **What does `Field(..., gt=0)` mean?**
   The ellipsis `...` indicates the field is mandatory, and `gt=0` specifies that the numeric value must be greater than zero.
